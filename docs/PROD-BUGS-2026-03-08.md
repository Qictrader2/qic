## BUG 1: Resell slider allows negative markup (discount below original offer)

**Severity:** HIGH — business logic violation  
**Status:** Code identified, not yet fixed

**Problem:**  
When reselling an offer, the markup slider allows values from -50% to +100%. If the original offer has a 1% premium, a reseller can set -50%, effectively selling at a massive discount below the original price. Resellers should only be able to *add* markup on top of the original offer's premium — never discount below it.

**Root cause:**  
`Frontend/src/app/(main)/offer/resell/_components/MarkupControls.tsx` hardcodes:

```
MARKUP_MIN = -50
MARKUP_MAX = 100
```

There is no `minMarkup` prop to let callers enforce a floor. Both `TraderCard.tsx` and `EscrowModal.tsx` use `MarkupControls` without any minimum override.

**Fix required:**
1. Add `minMarkup?: number` prop to `MarkupControls`
2. Use `minMarkup ?? MARKUP_MIN` as the slider min and in `validateMarkup`
3. In `TraderCard.tsx` and `EscrowModal.tsx` (resell context), pass `minMarkup={0}` so resellers can only add markup, never discount

**Files:**
- `Frontend/src/app/(main)/offer/resell/_components/MarkupControls.tsx` (lines 14-16, 79-82, 207-212)
- `Frontend/src/app/(main)/offer/resell/_components/TraderCard.tsx`
- `Frontend/src/app/(main)/offer/resell/_components/EscrowModal.tsx`

## BUG 2: Offer creation — "enter a valid price" + escrow wallet errors

**Severity:** HIGH — blocks offer creation  
**Status:** Code identified, not yet fixed

### 2a: "Please enter a valid price"

**Problem:**  
User tries to create an offer but gets a validation error "Please enter a valid price."

**Root cause:**  
`Frontend/src/utils/validation.ts` (line 165-169):
```
export function validatePrice(price: string): ValidationResult {
  const { isValid, value } = parseNumericInput(price)
  if (!isValid) return { isValid: false, error: "Please enter a valid price" }
  if (value <= 0) return { isValid: false, error: "Price must be greater than 0" }
}
```

The `parseNumericInput` function may be rejecting valid input formats (e.g. locale-formatted numbers, leading zeros, or edge cases). Need to verify what input the user entered. This could also be a market-based pricing mode issue where the price field is empty because the user expects it to be auto-calculated.

### 2b: "No escrow wallet available"

**Problem:**  
After placing an offer, user is told to fund escrow, then sees "no escrow wallet available."

**Root cause:**  
The backend creates an escrow record in the DB but escrow *wallet* creation requires blockchain service integration. The backend handlers in `qictrader-backend-rs/src/api/escrow.rs` (lines 606-630) create the escrow, but the wallet assignment path (`find_wallet_by_escrow_id`) returns None if no on-chain wallet was generated.

The offer was still created (user confirmed seeing it), but the escrow funding flow is broken because no wallet address is assigned.

**Files:**
- `Frontend/src/utils/validation.ts` (lines 165-190, 368-371)
- `Frontend/src/app/(main)/offer/create/page.tsx` (lines 261-277)
- `qictrader-backend-rs/src/api/escrow.rs` (lines 388-392, 606-630)
- `Frontend/src/services/offer-escrow.service.ts` (lines 85-169)
- `Frontend/src/app/(offers)/offer/[id]/fund-escrow/page.tsx` (line 388)

## BUG 3: Withdrawal gas fee estimation always fails (500)

**Severity:** MEDIUM — feature is non-functional but has fallback  
**Status:** Backend endpoint is a stub

**Problem:**  
Every withdrawal attempt shows "Failed to estimate gas fee. Using fallback estimate." The frontend retries every ~30 seconds, generating continuous 500s.

**Root cause:**  
The backend endpoint is a **hard-coded stub** that always throws:

```rust
// qictrader-backend-rs/src/api/gas.rs:330-333
pub async fn get_withdrawal_fee(auth: AuthUser) -> Result<Json<WithdrawalFeeResponse>> {
    let _ = auth;
    Err(AppError::internal("requires blockchain service"))
}
```

This returns 500 on every call. The frontend catches it and falls back to a hardcoded 5 USDT estimate (`Frontend/src/services/gas-api.ts` line 265-294).

**Prod log evidence:**  
`GET /api/v1/gas/withdrawal-fee?withdrawalAmount=21` → **status=500** — repeated every 30 seconds from IP `169.0.253.159`. At least 14 occurrences in the last 7 minutes of logs.

**Fix required:**  
Either implement the gas fee endpoint properly (query blockchain RPCs for actual gas prices) or return a proper static estimate instead of an error, so the frontend doesn't show error UI.

**Files:**
- `qictrader-backend-rs/src/api/gas.rs` (lines 330-333)
- `Frontend/src/services/gas-api.ts` (lines 101-130, 265-294)
- `Frontend/src/components/features/wallet/GasFeeDisplay.tsx` (lines 91-103)

## BUG 4 (from logs): Deposit monitor — blockchain RPCs rate-limited (429)

**Severity:** HIGH — deposit detection is broken for BTC, TRON, ETH  
**Status:** Ongoing in production

**Problem:**  
The deposit monitor job polls escrow wallet balances across all networks. Multiple blockchain RPCs are returning 429 (rate limited):

### BTC — Blockstream API
```
DEPOSIT_MONITOR: balance fetch failed network=bitcoin_mainnet
BTC API returned 429 Too Many Requests: "Blockstream Explorer API NOTICE:
Your request rate exceeds the current limit."
```
Blockstream is capping unauthenticated usage at 500k req/month and 700 req/min starting July 2025. The deposit monitor is burning through this limit by polling every wallet every cycle.

### TRON — TronGrid API  
```
DEPOSIT_MONITOR: balance fetch failed network=tron_mainnet crypto=USDT
TRON RPC returned 429: "request rate exceeded the allowed_rps(3)"
```
The free TRON RPC is limited to 3 requests/second. With multiple escrow wallets, the monitor exceeds this immediately.

### ETH — RPC parse failure
```
DEPOSIT_MONITOR: balance fetch failed network=ethereum_mainnet
ETH RPC parse failed: error decoding response body
```
ETH RPC is returning unparseable responses (possibly also rate-limited, returning HTML error pages instead of JSON).

**Impact:**  
Deposits to BTC, TRON (USDT), and ETH escrow wallets **will not be detected**. Users deposit funds but the system doesn't see them.

**Fix required:**
1. Add rate limiting/backoff to the deposit monitor (don't poll all wallets every cycle)
2. Use authenticated/paid API tiers for BTC (Blockstream) and TRON (TronGrid)
3. Fix ETH RPC endpoint — likely needs a working QuickNode endpoint configured

## BUG 6 (from code): `let _ = auth` in gas.rs — security issue

**Severity:** MEDIUM — violates auth rules  
**Status:** Code-level

Both `get_withdrawal_fee` and `validate_gas` in `gas.rs` discard the auth user:
```rust
let _ = auth;        // line 331
let _ = (auth, req); // line 326
```

Per the project's engineering rules, `let _ = auth` is a security vulnerability — the auth extractor runs but no authorization check happens. Even though these are stubs, they should either use the auth data or not require auth.

## BUG 7: Vercel frontend hitting old Node.js backend — signup/username check broken

**Severity:** CRITICAL — blocks all new signups  
**Status:** Env var fixed, awaiting confirmed deploy

**Problem:**  
The Vercel production frontend (`qictrader.com`) has `NEXT_PUBLIC_API_URL` baked in pointing to the **old, deprecated Node.js backend** (`https://qic-trader-backend-f44367dc781f.herokuapp.com/api/v1`) instead of the current Rust backend (`https://qictrader-backend-rs-13eab0516d9a.herokuapp.com/api/v1`).

This causes:
- **Username check** → old backend returns 500 → frontend interprets as "username taken"
- **Signup** → old backend returns 409 → "Username is already taken" even when it's free
- **Affiliate dashboard** → API calls fail → page is blank, no referral link shown
- **All authenticated API calls** → may hit wrong backend

**Root cause:**  
The `qictrader-frontend` Vercel project had a stale `NEXT_PUBLIC_API_URL` environment variable pointing to the old Node.js backend. This overrode the correct value in `.env.production` (which was committed with the right URL). The Vercel CLI `env ls` did not consistently show this variable, making diagnosis difficult.

**Console evidence:**
```
HttpClient initialized {"baseUrl":"https://qic-trader-backend-f44367dc781f.herokuapp.com/api/v1"}
GET https://qic-trader-backend-f44367dc781f.herokuapp.com/api/v1/users/check-username/jpsarfat 500
```

**Fix applied:**
1. Removed old `NEXT_PUBLIC_API_URL` from `qictrader-frontend` Vercel project
2. Re-added with correct Rust backend URL
3. Triggered fresh deploy

**Verification needed:**  
After deploy goes live, confirm browser console shows `baseUrl` pointing to `qictrader-backend-rs-*`.

**Files:**
